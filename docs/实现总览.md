# Agent Harness 最小实现总览（中文）

本文档说明当前仓库中已经实现的最小可运行 Agent Harness：
- agent 循环（async loop）
- tool-use（工具注册、分派、执行）
- sandbox（本地安全执行环境）
- API 与前端联调入口

目标是帮助你快速看懂这版实现的结构和运行路径，便于继续规划下一步。

## 1. 当前实现范围

已实现：
1. OpenAI 兼容 Chat Completions 调用链
2. 后端注册模型与默认模型机制（当前已注册 MiniMax M2.7）
3. 多轮 tool_call -> tool_result 回填循环
3. 本地 sandbox 文件/命令能力与越权拦截
4. FastAPI /chat、/chat/stream、/models 与 /conversations 接口
5. APP API 入口（APP/api/app.py）
6. SSE 流式响应接口（/chat/stream）
7. 线程级上下文治理（thread_id + JSON context store）
8. 完整会话历史持久化（conversation.json + messages.json）
9. skills 会话复用提示（pinned_skills + context_governance）
10. DeerFlow 风格 subagent 委派（task 工具 + 双线程池执行器）
11. subagent 生命周期流式事件（task_started/task_running/task_completed/task_failed/task_timed_out）
12. 内置知识库 RAG（Qdrant hybrid dense+sparse + rerank）
13. 最小测试覆盖（当前 20+ 个 RAG / tool / loop / service / conversation 测试通过）

暂未实现（有意留给后续迭代）：
1. skills/subagents/reflection/guardrails/mcp 的进一步扩展
2. 多租户会话管理与长期用户记忆
3. 更严格的命令白名单与审计
4. Electron 桌宠与 subagent 事件的直接联动桥接
5. 更细粒度的事件追踪与观测面板

## 3. 一次请求的完整路径

以“前端发送一句话”为例：

1. 前端触发发送
- [frontend/src/components/ChatContainer.tsx](../frontend/src/components/ChatContainer.tsx) 调用 sendMessage。

2. 前端请求后端
- [frontend/src/lib/api.ts](../frontend/src/lib/api.ts) 会先 GET /models 加载可选模型，再通过 /chat 或 /chat/stream 发起请求。

3. FastAPI 接收请求
- [APP/api/app.py](../APP/api/app.py) 的 chat()/chat_stream() 拿到 message + thread_id + messages，调用 ChatService。
- 会话侧接口 `GET /conversations`、`GET /conversations/{thread_id}`、`POST /conversations/create`、`POST /conversations/{thread_id}/rename` 同样由 `ChatService` 提供底层能力。

4. HarnessClient 组装运行时
- [harness/client.py](../harness/client.py) 初始化：
  - BasicSandbox
  - ToolRegistry（默认工具 + 可选 task 工具）
  - ToolExecutor
  - OpenAIChatCompletionsClient
  - AsyncAgentLoop
  - SubagentExecutor（按需由 task 工具触发）

4.1 上下文治理注入
- ChatService 会基于 thread_id 加载线程上下文（recent_messages/pinned_skills/compact_summary）。
- 若存在完整会话历史，则优先使用 `messages.json` 中最近窗口作为模型 history。
- HarnessClient 会将 pinned_skills/compact_summary 注入系统提示词。
- AsyncAgentLoop 会在 system 后注入历史消息，再拼接当前用户输入。

5. Agent 循环执行
- [harness/agents/loop.py](../harness/agents/loop.py) 的 AsyncAgentLoop.run()：
  - 第一步调用模型
  - 若无 tool_call，直接返回文本
  - 若有 tool_call，交给 ToolExecutor
  - 将 assistant/tool 消息回填给模型
  - 循环直到完成或 max_steps 到达

6. 工具层执行
- [harness/tools/executor.py](../harness/tools/executor.py) 分派到具体工具 handler。
- handler 在 [harness/tools/registry.py](../harness/tools/registry.py) 中定义。
- 当工具是 `task` 时，会进入 [harness/client.py](../harness/client.py) 的 `_run_subagent()`，再由 [harness/subagents/executor.py](../harness/subagents/executor.py) 用 `_scheduler_pool -> _execution_pool` 双线程池启动子 agent。
- 当工具是 `knowledge_search` 时，会进入 [harness/rag/service.py](../harness/rag/service.py) 的知识库检索链路：hybrid 召回后再做 rerank。

7. sandbox 落地执行
- [harness/sandbox/runtime.py](../harness/sandbox/runtime.py) 执行读写文件、列目录、替换字符串、bash。
- 所有路径都会经过 _resolve_path，确保不越出 sandbox 根目录。

8. 返回结果
- 后端返回 text + trace（steps/tool_calls/errors）。
- `/chat` 会额外返回 `thread_id`；`/chat/stream` 的 `done` 事件也会回传 `thread_id`。
- trace 现已包含 subagent_calls / subagent_errors。
- 流式场景返回 SSE 事件：start / delta / task_started / task_running / task_completed / task_failed / task_timed_out / done / error。
- 前端把 text 渲染成 assistant 消息。
- 前端通过下拉框切换模型，且只展示后端已注册模型。

### 3.1 当前 thread 目录结构

当前每个 thread 目录的核心结构为：

```text
.sandbox/threads/<normalized_thread_id>/
  context.json
  conversation.json
  messages.json
  workspace/
  uploads/
  outputs/
  skills/
```

职责划分：
1. `context.json`：模型轻量上下文与 compact 结果
2. `conversation.json`：会话摘要元数据
3. `messages.json`：完整 user / assistant 消息历史
4. `workspace/uploads/outputs/skills`：线程工作区与文件资产

## 4. 最小闭环里的三个关键点

### 4.1 Agent Loop（最小）

在 [harness/agents/loop.py](../harness/agents/loop.py) 中实现了最小循环协议：
1. 输入 user prompt
2. 调模型拿输出
3. 解析 function_call
4. 执行工具
5. 回填 assistant/tool 消息
6. 继续下一轮

停止条件：
1. 本轮无 tool_call
2. 超过 max_steps

### 4.2 Tool-use（最小）

当前默认主 agent 内置 9 个基础工具（见 [harness/tools/registry.py](../harness/tools/registry.py)）：
1. bash
2. ls
3. read_file
4. write_file
5. str_replace
6. browser_search
7. browser_open
8. knowledge_search
9. web_search

当主 agent 开启 subagent 能力时，还会额外注册：
10. task

执行保障：
1. 未知工具返回标准错误，不会崩主循环
2. 工具异常被捕获并写入 ToolResult.is_error
3. trace.errors 会累计错误次数
4. `task` 工具默认禁止嵌套 subagent
5. 每轮最多允许 3 个 `task` 调用，超出的会被截断

### 4.4 Subagent（当前版本）

当前实现对齐 DeerFlow 的几个关键原则：
1. 主 agent 负责拆分、委派、汇总
2. subagent 通过 `task` 工具启动
3. subagent 使用独立 prompt 和受限工具集合
4. subagent 与主 agent 共享 thread workspace，但不继承历史对话
5. 执行器使用 `_scheduler_pool -> _execution_pool` 双线程池

当前内置 subagent：
1. `general-purpose`
2. `bash`

### 4.3 Sandbox（最小）

见 [harness/sandbox/runtime.py](../harness/sandbox/runtime.py)：
1. root_dir 作为执行根目录
2. 绝对/相对路径统一 resolve
3. 如果目标路径不在 root_dir 下，抛 SandboxPermissionError
4. 高风险命令（如 `rm -rf`、`sudo`、关机/格式化类命令）会在执行前被拦截
5. 命令输出和文件读取有长度截断，避免上下文爆炸

## 8. 多模型自由选择扩展方案（规划）

目标：让前端/调用方可以自由选择模型，同时保证后端在“可用性、可观测性、可维护性”三方面稳定。

### 8.1 配置层目标（harness）

当前状态：配置层只从一组环境变量组装 1 个 RegisteredModel。

扩展方向：
1. 模型注册改为“多条目列表”，每个条目至少包含：
- name（前端和 API 选择用）
- display_name（展示名）
- provider（openai-compatible 等）
- model（上游模型 ID）
- base_url（每个模型可独立）
- api_key_env（每个模型可独立）
- temperature / max_tokens（每模型默认参数）
- supports_vision（能力标签）
2. 默认模型仍保留为 default_model，但语义仅为“未指定 model 时的兜底值”。
3. 配置加载时增加一致性校验：
- default_model 必须存在于 models 列表
- name 不能重复
- 必填字段不能为空
- api_key_env 必须指向有效环境变量名（不是密钥值本身）

### 8.2 API 层目标（APP/api）

扩展方向：
1. 保持 /chat 请求结构不变，通过 model 字段显式选择。
2. /models 返回完整模型清单，并明确 default 标记。
3. 对非法 model 名称返回 400（而不是 500），保证错误可预期。
4. Swagger 中将 model 字段描述为“可选，缺省走 default_model”。

### 8.3 Service 层目标（APP/service）

扩展方向：
1. Service 负责业务策略，不直接硬编码模型路由规则。
2. 路由优先级建议固定：
- 请求显式 model
- 默认模型 default_model
3. 若未来有租户/用户偏好，可在 Service 增加“业务级覆盖层”，但最终仍调用 harness 的统一 get_model 逻辑。

### 8.4 运行时目标（harness/client）

扩展方向：
1. 保持“共享基础设施复用 + 按请求选择模型”的模式：
- 复用：sandbox、tools、executor
- 按请求变化：模型 client、loop settings
2. 可选优化：增加按 model_name 的轻量缓存，减少高频请求下重复构建开销。
3. 约束：缓存只能缓存“无会话状态对象”，避免跨请求污染上下文。

### 8.5 前端联动目标

扩展方向：
1. 启动时从 /models 拉取候选并展示。
2. 每次发送时把当前选中 model 传给 /chat。
3. 若模型下线或不可用，前端提示并回退到 default。

### 8.6 可观测性与运维建议

扩展方向：
1. 结构化日志至少记录：request_id、model_name、provider、latency、error_type。
2. 监控按模型维度聚合成功率和耗时，便于比较不同模型质量与成本。
3. 灰度发布时先增加模型再切默认，避免一次性切换带来风险。

### 8.7 迁移顺序建议

建议分阶段推进，降低风险：
1. 先支持“多模型配置解析 + /models 正确返回”。
2. 再打通 /chat 的显式 model 选择与 400 错误语义。
3. 最后做缓存与监控增强。

这样可以先保证功能可用，再逐步做性能与运维完善。
