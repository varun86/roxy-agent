## 1. 这次改造的核心目标

这次改造不是“简单多加一个 agent”，而是把原先的单 agent harness 扩展成：
1. 主 agent 负责总控、拆分任务、汇总结果
2. `task` 工具负责作为唯一 subagent 委派入口
3. subagent 在独立 prompt 和受限工具集下执行
4. 主线程通过 SSE 接收 subagent 生命周期事件
5. 后端用 DeerFlow 风格双线程池执行 subagent

## 2. 用户请求下来后的完整链路

### 2.1 主入口

请求从 [APP/api/app.py](../APP/api/app.py) 进入后：
1. `POST /chat` 或 `POST /chat/stream` 收到用户输入
2. [APP/service/chat_service.py](../APP/service/chat_service.py) 恢复 thread 上下文
3. `ChatService` 调用 `HarnessClient.run_async(...)`
4. `HarnessClient` 组装主 agent 并进入 [harness/agents/loop.py](../harness/agents/loop.py)

### 2.2 主 agent 如何决定是否拆分

主 agent 不会被后端硬编码成“遇到问题就开 subagent”。  
真正的拆分决策发生在模型这一层，依据是 [harness/agents/prompt.py](../harness/agents/prompt.py) 注入的 `<subagent_system>` 提示：
1. 复杂任务可以拆成多个并行子任务
2. 每轮最多发 3 个 `task`
3. 简单任务不要用 `task`
4. 超过 3 个子任务时必须分批

因此当用户请求较复杂时，主 agent 可能在一轮响应中输出多个 `task(...)` tool call，而不是直接生成最终文本。

### 2.3 主循环如何处理 `task`

在 [harness/agents/loop.py](../harness/agents/loop.py) 中：
1. 主 agent 拿到模型输出的 `tool_calls`
2. `_truncate_subagent_calls()` 会先限制本轮最多 3 个 `task`
3. `ToolExecutor.execute_batch()` 并发执行这一轮工具调用
4. 普通工具直接返回结果
5. `task` 工具会进入 subagent 执行链路
6. 所有 tool result 回填给主 agent
7. 主 agent 再决定是继续发下一批 `task`，还是综合回答

这意味着“拆分”由模型完成，“调度”由 loop 和 executor 完成，“汇总”仍由主 agent 完成。

## 3. `task` 工具与 subagent 委派链路

### 3.1 `task` 是唯一委派入口

[harness/tools/registry.py](../harness/tools/registry.py) 中注册了 `task` 工具。  
它接收：
1. `description`
2. `prompt`
3. `subagent_type`
4. `max_steps`

它本身不直接执行工作，而是调用 `ToolRuntime.run_subagent(...)`。

### 3.2 为何禁止嵌套 subagent

`task` 工具会检查 `runtime.context.subagent_depth`：
1. `subagent_depth == 0` 代表当前是主 agent，可以委派
2. `subagent_depth > 0` 代表当前已经在 subagent 内，禁止再调用 `task`

这与 DeerFlow 在设计上的“单层委派”原则一致，避免递归代理带来的复杂性和失控。

### 3.3 subagent 如何运行

[harness/client.py](../harness/client.py) 的 `_run_subagent(...)` 负责：
1. 根据 `subagent_type` 从 [harness/subagents/registry.py](../harness/subagents/registry.py) 取配置
2. 构造 `task_id`
3. 向前端发 `task_started`
4. 创建 `SubagentExecutor`
5. 把真实执行逻辑包装成 `run_callable`
6. 用 `SubagentExecutor.execute_async()` 启动后台执行
7. 轮询子任务状态
8. 把完成/失败/超时状态转换成 SSE 事件和 tool result

主 agent 最终看到的只是一个普通工具结果，例如：
- `Task Succeeded. Result: ...`

所以 subagent 对主 loop 来说仍然是标准 tool-use 的一部分。

## 4. DeerFlow 风格双线程池是如何落地的

核心文件是 [harness/subagents/executor.py](../harness/subagents/executor.py)。

### 4.1 两层线程池职责

当前实现保留了 DeerFlow 的 `_scheduler_pool -> _execution_pool` 结构：

1. `_scheduler_pool`
- 负责后台任务调度
- 负责把任务状态从 `PENDING` 切换到 `RUNNING`
- 负责等待执行结果和处理超时

2. `_execution_pool`
- 负责真正运行 subagent
- 在独立线程中调用 `execute()`
- `execute()` 再通过 `asyncio.run(_aexecute(...))` 启动一个完整的异步 subagent agent loop

### 4.2 为什么不是直接 `await`

使用双线程池的原因：
1. 更贴近 DeerFlow 的后台任务模型
2. 调度与执行分离，超时管理更清晰
3. 方便后续扩展为长期后台任务或更复杂观测
4. 不把 subagent 生命周期和主请求协程直接耦死

### 4.3 运行状态

当前支持的状态：
1. `PENDING`
2. `RUNNING`
3. `COMPLETED`
4. `FAILED`
5. `TIMED_OUT`

`_run_subagent(...)` 会轮询这些状态，并发出：
1. `task_started`
2. `task_running`
3. `task_completed`
4. `task_failed`
5. `task_timed_out`

## 5. subagent 与主 agent 共享了什么，隔离了什么

共享：
1. 同一个 thread sandbox 根目录
2. 同一个 `workspace_dir`
3. 同一个 thread 文件上下文

隔离：
1. 不继承主 agent 的历史消息
2. 使用 subagent 自己的 system prompt
3. 使用更严格的工具 allowlist / denylist
4. 默认关闭 `task`，禁止再次委派

这意味着 subagent 可以读写与主 agent 相同的线程工作区，但不会把自己的推理上下文膨胀到主对话历史里。

## 6. `client.py` 到底在干什么

[harness/client.py](../harness/client.py) 最容易看晕的原因，是它同时承担了：
1. 主 agent 装配
2. subagent 装配
3. sandbox 构建
4. tool runtime 构建
5. skills 注入
6. subagent 生命周期桥接

最实用的理解方式是：把它当成“agent 工厂”。

### 6.1 `run_async(...)`

这是给 service 层调用的统一入口：
1. 调 `_build_agent(...)` 组装主 agent
2. 根据是否需要流式输出，调用 `run()` 或 `run_with_stream()`

### 6.2 `_build_agent(...)`

这是整个文件最核心的函数。它负责：
1. 选择模型
2. 构造 sandbox
3. 创建 `ToolRegistry`
4. 决定是否注册 `task`
5. 根据 allowlist / denylist 过滤工具
6. 创建 `RuntimeContext`
7. 创建 `ToolRuntime`
8. 创建 `ToolExecutor`
9. 生成系统提示词
10. 创建 `AsyncAgentLoop`

它的本质是把“配置 + prompt + 工具 + runtime context”拼成一个可运行 agent。

### 6.3 `_make_runtime_context(...)`

它把工具执行真正需要的上下文统一封装起来：
1. `thread_id`
2. `thread_root`
3. `workspace_dir`
4. `model_name`
5. `subagent_depth`
6. `max_subagents`
7. `subagent_timeout_seconds`

因此工具层无需直接依赖 `HarnessClient` 本身，只需消费 `ToolRuntime.context`。

### 6.4 `_run_subagent(...)`

这个函数把“subagent 是工具”这件事桥接起来：
1. 主 agent 调 `task`
2. `task` 调 `_run_subagent(...)`
3. `_run_subagent(...)` 再用 `_build_agent(...)` 创建一个子 agent
4. 子 agent 被丢进 `SubagentExecutor`
5. 子 agent 的结果再变回主 agent 可见的 tool output

可以把它理解成：
- `client.py` 先创建主 agent
- 主 agent 又通过 `client.py` 间接创建 subagent

所以 `client.py` 是“装配层”和“递归创建 agent 的工厂”。

## 7. 当前内置的 subagent 类型

当前定义在 [harness/subagents/registry.py](../harness/subagents/registry.py)：

1. `general-purpose`
- 通用型 subagent
- 适合探索、分析、多步骤任务
- 允许较宽的工具集合

2. `bash`
- 偏命令执行型 subagent
- 适合输出较长、步骤较多的 shell 工作
- 只允许 `bash/ls/read_file/write_file/str_replace`

## 9. 当前版本的设计边界

当前已经实现：
1. 单层 subagent 委派
2. DeerFlow 风格双线程池
3. 主 loop 中的 `task` 限流
4. SSE 子任务生命周期事件
5. 线程级共享 workspace

当前暂未实现：
1. subagent 嵌套
2. 更丰富的 subagent 类型注册机制
3. 前端树状可视化 subagent 面板
4. Electron 桌宠与 subagent 事件的直接联动桥接

如果后续继续扩展，优先级最高的下一步通常是：
1. 将 SSE 的 `task_*` 事件桥接到 desktop 端 `SubagentStart/SubagentStop`
2. 为 subagent 增加更细粒度的运行日志展示
3. 为 `client.py` 再拆一层 runtime builder，降低装配复杂度
