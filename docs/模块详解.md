# 模块详解（逐文件）

本文按你当前实现逐文件解释，避免抽象空谈。

## 1. 配置层

文件：
- [harness/config/settings.py](../harness/config/settings.py)

### 1.1 OpenAIConfig

作用：保存模型调用配置。
字段：
1. model
2. base_url
3. api_key_env

关键方法：
- read_api_key()
  - 从 api_key_env 对应的环境变量读取 key。
  - 为空则抛 ValueError，防止无密钥运行。

### 1.2 SandboxConfig

作用：定义 sandbox 根目录和命令超时。

### 1.3 RuntimeConfig

作用：控制循环步数和输出截断。
当前还额外控制：
1. 是否启用 subagent
2. 每轮最大并发 subagent 数量
3. subagent 默认超时时间

### 1.4 load_harness_config()

作用：从环境变量生成 HarnessConfig。
要点：
1. 支持默认值，便于最小启动
2. sandbox root 会 resolve 成绝对路径

## 2. 数据模型层

文件：
- [harness/models/types.py](../harness/models/types.py)

### 2.1 ToolCall

表示模型发起的一次工具调用：
1. id
2. name
3. arguments

### 2.2 ToolResult

表示工具执行结果：
1. call_id（与 ToolCall.id 对齐）
2. output
3. is_error

### 2.3 AgentTrace

运行轨迹统计：
1. steps
2. tool_calls
3. errors
4. subagent_calls
5. subagent_errors

### 2.4 AgentRunResult

最终返回给调用方的数据：
1. text
2. trace

### 2.5 RuntimeContext

作用：把工具运行时需要的线程上下文集中封装。
核心字段：
1. thread_id
2. thread_root
3. workspace_dir
4. model_name
5. subagent_depth
6. max_subagents
7. subagent_timeout_seconds

## 3. Sandbox 层

文件：
- [harness/sandbox/runtime.py](../harness/sandbox/runtime.py)

### 3.1 异常体系

1. SandboxError：基类
2. SandboxPermissionError：路径越权
3. SandboxExecutionError：执行失败

### 3.2 BasicSandbox 初始化

构造参数：
1. root_dir
2. command_timeout_seconds
3. max_output_chars

初始化时会创建 root_dir。

### 3.3 路径安全核心：_resolve_path()

逻辑：
1. 把相对/绝对路径统一 resolve
2. 检查 resolved 是否在 root_dir 下
3. 不在则抛 SandboxPermissionError

这是当前最关键的安全边界。

### 3.4 能力方法

1. run_bash(command)
- 在 root_dir 内执行 shell 命令
- 捕获 stdout+stderr
- 非 0 退出码仍返回可读错误文本

2. list_dir(path)
- 列目录内容
- 不存在则抛 SandboxExecutionError

3. read_file(path, start_line, end_line)
- 支持整文件读取和行区间读取

4. write_file(path, content, append)
- 覆盖或追加
- 自动创建父目录

5. str_replace(path, old_str, new_str, replace_all)
- 单次或全量替换
- old_str 为空或不存在时抛错

## 4. Tool 层

文件：
- [harness/tools/registry.py](../harness/tools/registry.py)
- [harness/tools/executor.py](../harness/tools/executor.py)

### 4.1 ToolRegistry

职责：
1. 注册工具规范 ToolSpec
2. 注册 handler
3. 输出模型需要的 tool schema
4. 根据 allowlist / denylist 派生受限工具集合

新增点：
1. handler 现在接收 `ToolRuntime`
2. `ToolRuntime` 中包含 sandbox、RuntimeContext、事件发射器、subagent 启动器
3. 主 agent 可按需注册 `task` 工具

with_default_tools() 中把 sandbox 能力映射成异步工具。
当 `include_task_tool=True` 时，还会注册 subagent 委派工具 `task`。

### 4.2 ToolExecutor

职责：
1. 按 name 找 handler
2. 执行单个工具调用
3. 执行批量工具调用

容错策略：
1. 未知工具 -> is_error=True
2. handler 抛异常 -> is_error=True

优势：不会把主循环直接打崩。
当前 `execute_batch()` 已改为并发执行，这样同一轮多个 `task` 可以并行委派。

## 5. Subagent 层

文件：
- [harness/subagents/config.py](../harness/subagents/config.py)
- [harness/subagents/registry.py](../harness/subagents/registry.py)
- [harness/subagents/executor.py](../harness/subagents/executor.py)

### 5.1 SubagentConfig

描述一个 subagent 的最小配置：
1. name
2. description
3. system_prompt
4. tools
5. disallowed_tools
6. model
7. max_steps
8. timeout_seconds

### 5.2 内置 subagent

当前内置：
1. `general-purpose`
2. `bash`

前者偏通用探索与多步骤任务，后者偏命令执行。

### 5.3 SubagentExecutor

这是 DeerFlow 风格的双线程池执行器。

包含两层线程池：
1. `_scheduler_pool`
- 负责后台调度、状态切换、超时等待
2. `_execution_pool`
- 负责真正运行 subagent

执行方式：
1. `execute_async()` 先把任务放入 `_scheduler_pool`
2. scheduler 再把 `execute()` 提交给 `_execution_pool`
3. `execute()` 内部通过 `asyncio.run(_aexecute())` 在独立线程中运行子 agent

支持状态：
1. PENDING
2. RUNNING
3. COMPLETED
4. FAILED
5. TIMED_OUT

## 6. Agent 循环层

文件：
- [harness/agents/loop.py](../harness/agents/loop.py)

### 5.1 ChatCompletionsModelClient（协议）

定义模型客户端最小接口：
- create_response(messages, tools, temperature, max_tokens)

作用：让循环层依赖“协议”而不是某个具体 SDK。

### 5.2 OpenAIChatCompletionsClient

职责：
1. 调用 AsyncOpenAI.chat.completions.create
2. 解析 message.tool_calls
3. 把 arguments JSON 解析成 dict
4. 返回 (output_text, tool_calls)

### 5.3 AsyncAgentLoop.run()

主循环步骤：
1. 初始化 messages 为 system + user prompt
2. 调 model_client.create_response()
3. 若无 tool_calls：直接返回
4. 若有 tool_calls：调用 tool_executor.execute_batch()
5. 构造 assistant/tool 消息写回 messages
6. 继续下一轮
7. 超 max_steps 时返回带提示的结果

关键点：
1. 通过完整 messages 历史维持 Chat Completions 的多轮上下文
2. 通过 trace 累计步骤/调用/错误
3. 当前会在执行前截断超出上限的 `task` 工具调用
4. 当前会单独统计 subagent_calls / subagent_errors

## 7. 组装层

文件：
- [harness/client.py](../harness/client.py)

### 6.1 HarnessClient

把所有模块装配起来：
1. 读取配置
2. 创建 BasicSandbox
3. 创建 ToolRegistry + ToolExecutor
4. 创建 OpenAIChatCompletionsClient
5. 创建 AsyncAgentLoop
6. 按需构造 subagent runtime 与 SubagentExecutor

对外能力：
1. run_async(prompt)
2. run(prompt)

### 6.2 `client.py` 的关键方法

1. `_make_sandbox()`
- 根据 thread_paths 决定 sandbox 根目录和工作目录

2. `_make_runtime_context()`
- 构造工具层要消费的 RuntimeContext

3. `_build_agent()`
- 这是整个文件最核心的工厂方法
- 负责把模型、sandbox、registry、runtime、prompt、loop 全部装起来
- 主 agent 和 subagent 都通过它创建

4. `_run_subagent()`
- 负责把 `task` 工具调用桥接成真实子 agent 执行
- 会发出 `task_started/task_running/task_completed/...` 事件
- 会轮询 SubagentExecutor 中的后台结果

5. `run_async()`
- 给 service 层的统一入口
- 根据是否流式输出调用 `run()` 或 `run_with_stream()`

### 6.3 为什么 `client.py` 看起来最复杂

因为它同时承担了：
1. 主 agent 装配
2. subagent 装配
3. tools runtime 注入
4. skills prompt 注入
5. 事件桥接
6. 后台执行协调

最准确的心智模型是：`client.py` 不是业务逻辑层，而是运行时“总装工厂”。

## 11. 已知边界与设计取舍

当前是“最小可跑版”，主要取舍如下：
1. 强调可读和可扩展，不追求一次到位的安全策略
2. 先用本地 sandbox，不引入容器/远端隔离
3. tool schema 手写，后续可迁移到 pydantic 自动生成
4. 当前只实现单层 subagent，不支持嵌套
5. 采用 DeerFlow 风格双线程池，但暂未扩展为更完整的后台任务观测系统

这些取舍是为了先保证最小闭环跑通，便于你快速进入下一阶段迭代。
