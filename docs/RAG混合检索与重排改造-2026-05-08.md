# RAG 混合检索与重排改造（2026-05-08）

本文记录 2026-05-08 对当前 RAG 子系统的一次重要升级，目标是把原先的“单路 dense 检索”升级为“Qdrant hybrid 检索 + rerank”的两阶段方案，便于后续排查召回质量、索引结构和运行方式。

## 1. 改造背景

在初版实现中，知识库检索链路只有一条 dense 路线：

1. 文档切块
2. 用 `BAAI/bge-small-zh-v1.5` 生成 dense embedding
3. 写入 Qdrant
4. 查询时仅做 dense 相似度召回

这个版本能跑通最小闭环，但在以下场景中效果不够理想：

1. 专有名词、角色名、篇章名、术语等关键词问题
2. 用户提问和原文表达方式差异较大的问题
3. 候选召回后排序不稳定，容易把语义相近但不够精确的 chunk 放在前面

因此本次改造分为两层：

1. 第一阶段：Qdrant 内部 hybrid 检索（dense + sparse）
2. 第二阶段：cross-encoder rerank

## 2. 当前方案总览

当前知识库检索链路如下：

1. 文档加载与切块：
- 固定读取 `docs/knowledge/`
- 仍使用现有 chunk 逻辑，不在本轮改 chunk 策略

2. 建索引：
- dense 模型：`BAAI/bge-small-zh-v1.5`
- sparse 模型：`Qdrant/bm25`
- 每个 chunk 同时写入 dense 向量和 sparse 向量

3. 查询：
- 为 query 同时生成 dense query vector 和 sparse query vector
- 在 Qdrant 中使用 `query_points(prefetch=[...], query=FusionQuery(...))`
- 采用 `RRF` 融合 dense / sparse 两路召回结果

4. 重排：
- 对 hybrid 候选结果使用 `BAAI/bge-reranker-base` 做 cross-encoder rerank
- 最终把 rerank 后的结果返回给 `knowledge_search`

## 4. 索引结构变化

### 4.1 旧结构

旧版 Qdrant collection 只有一个匿名 dense 向量：

1. point id
2. vector: `list[float]`
3. payload: `chunk_id/source_path/title/text/updated_at`

### 4.2 新结构

新版 collection 改为 named vectors：

1. dense named vector：`dense`
2. sparse named vector：`sparse`
3. payload 仍保留：
- `chunk_id`
- `source_path`
- `title`
- `text`
- `updated_at`

Qdrant collection 创建时：

1. `vectors_config["dense"] = VectorParams(...)`
2. `sparse_vectors_config["sparse"] = SparseVectorParams(modifier=Modifier.IDF)`

这意味着：

1. 旧的单向量 collection 与新查询逻辑不兼容
2. 本次升级后必须全量重建知识库索引

## 5. 为什么必须重建索引

原因不是文档内容变化，而是“索引 schema 变化”。

当前 `index_documents()` 会：

1. 为每个 chunk 生成 dense vectors
2. 为每个 chunk 生成 sparse vectors
3. 调用 `store.upsert(chunks, dense_vectors, sparse_vectors)`

因此旧数据不能直接复用。

当前实现采用“重建型升级”：

1. `reindex-kb` 执行时会重建 collection
2. 若 collection 已存在，会先删除旧 collection
3. 再按新 schema 创建并全量写入

也就是说，升级后不需要手动删旧数据，但必须重新执行索引命令。

## 6. Hybrid 检索设计

### 6.1 为什么使用 Qdrant 内部 hybrid

本次没有额外引入独立 BM25 服务，而是直接使用：

1. Qdrant sparse vectors
2. `Qdrant/bm25` sparse embedding
3. Qdrant Query API 的 `prefetch + FusionQuery`

这样做的好处：

1. 部署简单，不新增独立检索组件
2. dense 与 sparse 共用同一个 point / payload
3. 查询链路更短，更容易调试和回溯

### 6.2 查询流程

查询时的逻辑：

1. dense query vector
- 用 `BAAI/bge-small-zh-v1.5` 生成

2. sparse query vector
- 用 `Qdrant/bm25` 生成

3. Qdrant hybrid query
- sparse prefetch 使用 `using="sparse"`
- dense prefetch 使用 `using="dense"`
- 主查询使用 `FusionQuery(fusion=Fusion.RRF)`

默认参数：

1. dense prefetch limit：20
2. sparse prefetch limit：20
3. hybrid candidate limit：10

### 6.3 为什么先用 RRF

本次默认选择 `RRF`，原因：

1. 对首版实现更稳
2. 不需要手动做分数归一化
3. 更适合先把 dense / sparse 双路召回闭环跑通

当前也在配置层预留了 `fusion_strategy`，后续可以切换到 `DBSF` 做对比实验。

## 7. Rerank 设计

### 7.1 初版为什么需要 rerank

Hybrid 召回解决的是“找回来更多正确候选”，但并不保证“正确候选一定排在最前面”。

因此本次加入第二阶段 rerank：

1. 先做 hybrid 召回
2. 对 top-N 候选做 cross-encoder 打分
3. 按 rerank score 重新排序

### 7.2 当前 rerank 选型

默认模型：

1. `BAAI/bge-reranker-base`

选择原因：

1. 中英文都可用
2. 本地部署路径清晰
3. 与当前 dense 模型风格一致，适合作为首版默认值

### 7.3 回退策略

rerank 不应该把整个检索链路打崩，因此当前实现包含兜底：

1. 若 reranker 初始化失败
2. 或 rerank 推理报错

则直接回退到 hybrid 原始排序，不中断 `knowledge_search`。

## 9. Tool-use 侧行为变化

工具名没有变化，仍然是：

1. `knowledge_search`

但工具语义增强了：

1. 更强调专有名词、设定、FAQ、制度、产品资料等问题优先检索
2. 更强调“先检索，再决定是否不知道”
3. 明确把检索结果视为证据数据，而不是可执行指令

工具输出也做了增强，便于调试：

1. `hybrid_score`
2. `rerank_score`

这让后续排查“为什么没命中”或“为什么排序怪异”更方便。

## 10. 测试覆盖

本次测试覆盖了四类重点：

1. collection schema
- 验证 Qdrant 使用 named dense + sparse vectors

2. hybrid 查询
- 验证 hybrid 检索能够返回预期 chunk
- 验证查询时实际使用 `RRF`

3. rerank
- 验证候选顺序会被 rerank 改写
- 验证 rerank 异常时能自动回退

4. tool-use 集成
- 验证 `knowledge_search` 输出包含 hybrid/rerank 分数
- 验证 agent loop 中知识库工具调用的 roundtrip 正常

## 11. 当前已知限制

当前版本仍有这些限制：

1. chunk 策略仍较简单，未针对中文叙事文本进一步优化
2. rerank 未做缓存，候选较多时会增加延迟
3. `Qdrant/bm25` 更偏关键词精确召回，不等同于稀疏语义增强模型
4. 还没有引入 ColBERT / late interaction
5. 还没有做召回质量评测脚本与离线 benchmark

## 12. 后续可继续迭代的方向

若后续继续追求效果，可考虑：

1. sparse 从 `Qdrant/bm25` 升级为 `bm42` 或 `minicoil`
2. rerank 从 cross-encoder 升级到 ColBERT late interaction
3. 针对中文文本优化 chunk 大小、重叠和标题拼接策略
4. 增加知识库命中评测集与离线 benchmark
5. 为 `knowledge_search` 增加结构化调试输出或 trace
